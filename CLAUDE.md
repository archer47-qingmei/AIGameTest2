# AIGameTest2 项目规范

## 架构原则

本项目遵循以下四条架构原则，**所有新功能设计和代码审查必须对照执行**：

### 1. 通过接口定义小模块
每个模块通过明确的公开接口（方法签名、信号）与外部通信，内部实现对外不可见。新增模块前先定义接口，再写实现。

### 2. 每个小模块只关注自己的功能
- **逻辑层**（`RefCounted` 子类）：只持有游戏状态和规则，不引用任何 Node 或 Scene
- **UI 层**（`Control`/`Node` 子类）：只做显示和事件转发，不持有任何游戏逻辑
- 违反此原则的代码必须在设计阶段拆分，不得进入实现

### 3. 剥离出可自动测试的逻辑（重要）
- 所有游戏逻辑必须写在 `RefCounted` 子类中，使其可在不启动 Godot 编辑器的情况下被实例化和测试
- 禁止将可测试逻辑写进 Node 脚本（`_ready`、`_process` 等生命周期函数不应含业务逻辑）
- 新增复杂逻辑模块时，同步考虑是否需要引入 GUT 编写单元测试

### 4. 依赖接口而非实现
- 模块间通过**信号**（`signal`）解耦，上层不直接调用下层的内部方法
- 数据通过 **Resource 子类**（`.tres` 文件）传递，不在代码中硬编码数值
- 新功能扩展应通过新增数据文件实现，而非修改逻辑代码

---

## 当前架构示意

```
UI 层（Node/Control）
  CombatScreen.gd       ← 只做显示，监听信号，转发按钮事件

逻辑层（RefCounted）
  CombatEngine.gd       ← 游戏状态机，发出 state_changed / combat_ended 信号
  EffectResolver.gd     ← 纯静态效果执行器，无状态
  Combatant.gd          ← 战斗者运行时状态

数据层（Resource / .tres）
  CardData.gd           ← 卡牌蓝图
  CardEffectData.gd     ← 单条效果蓝图
  EnemyData.gd          ← 敌人蓝图
  EnemyActionData.gd    ← 单条敌人行动蓝图
  data/cards/*.tres     ← 卡牌数据文件
  data/enemies/*.tres   ← 敌人数据文件
```

## 开发约定

- **语言**：与用户交流始终使用中文
- **版本节奏**：每个版本只做一件事（纯重构 / 纯新功能 / 纯数据），不混合
- **提交粒度**：每个独立变更单独提交，commit message 使用 `feat:` / `refactor:` / `docs:` 前缀
- **数据外置**：卡牌、敌人、关卡等所有内容数值必须存在 `.tres` 文件中，逻辑代码不得硬编码
- **开发日志**：每版本完成后输出到 `E:\MyWork\Devlog\`，命名格式：`{项目名}_{版本号}_{日期}.md`
