# v0.23.0 地图扩展 设计文档

**日期：** 2026-04-30
**版本：** v0.23.0
**主题：** 随机地图生成，4 列布局

---

## 目标

将当前固定 3 列 5 节点地图扩展为随机生成的 4 列 7 节点地图，每局布局不同，增加重玩价值。

---

## 地图结构

```
第0列(2节点)   第1列(2节点)   第2列(2节点)   第3列(1节点)
  COMBAT ─┐                                 
          ├──→ ??? ──┐                       
  COMBAT ─┘          ├──→ ??? ──┐            
                ??? ─┘          ├──→ BOSS
                                ??? ─┘
```

连接规则：第 N 列所有节点均连接到第 N+1 列所有节点。

---

## 生成规则（按列）

| 列 | 节点数 | 类型分配 | 敌人分配 |
|---|---|---|---|
| 第 0 列 | 2 | COMBAT × 2 | jaw_worm 和 fire_lizard 随机分配到 row 0 / row 1 |
| 第 1 列 | 2 | 1 COMBAT + 1 REST，随机分配行 | COMBAT → jaw_worm / fire_lizard 随机选一 |
| 第 2 列 | 2 | 从 {COMBAT, REST, ELITE} 随机选 2 种不同类型，随机分配行 | COMBAT → 随机选一，ELITE → elite_guard |
| 第 3 列 | 1 | BOSS（固定，row 0） | boss |

第 2 列三种可能组合：COMBAT+REST、COMBAT+ELITE、REST+ELITE（各 1/3 概率）。

## 架构变化

### 新增：MapGenerator.gd

```
class_name MapGenerator
extends RefCounted
```

静态工具类，职责：根据生成规则构造 `Array[NodeData]`，设置每个节点的 config（动态创建 NodeConfig 实例）和 connections。

公开接口：
```gdscript
static func generate() -> Array[NodeData]
```

NodeConfig 实例在 generator 内动态创建（`NodeConfig.new()`），不需要为每种组合新建 .tres 文件。

### 修改：GameManager.gd

`start_new_run()` 中移除硬编码的 5 个节点和连接关系，替换为：

```gdscript
var nodes: Array[NodeData] = MapGenerator.generate()
player_state.map_all_nodes.assign(nodes)
player_state.available_nodes.assign(nodes.filter(func(n): return n.config.column == 0))
```

其他方法（`is_final_node`、`is_elite_node`、`end_combat` 等）无需修改，逻辑基于节点属性而非硬编码位置。

### 修改：map/MapScreen.gd

`_get_node_pos()` 新增第 3 列坐标映射（在现有 col 0/1/2 基础上加 col 3 的 x 值）。行坐标不变。

## 验收标准

| 场景 | 预期行为 |
|---|---|
| 开始新局 | 地图显示 4 列，共 7 个节点 |
| 第 0 列 | 始终为 2 个 COMBAT 节点，敌人为 jaw_worm 和 fire_lizard（行顺序随机）|
| 第 1 列 | 始终为 1 个 COMBAT + 1 个 REST（行顺序随机）|
| 第 2 列 | 2 个节点，类型组合为 COMBAT+REST、COMBAT+ELITE、REST+ELITE 之一 |
| 第 3 列 | 始终为 1 个 BOSS 节点 |
| 多次开局 | 第 2 列节点类型在多局中出现不同组合 |
| 精英/Boss 遗物流程 | 与 v0.22.0 一致，is_elite_node / is_final_node 逻辑不变 |
| 休息站流程 | 第 1 列 REST 节点可正常进入休息界面 |
